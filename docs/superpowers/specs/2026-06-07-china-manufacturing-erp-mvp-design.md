# 中国中小制造企业开源 ERP MVP 方案

日期：2026-06-07

## 1. 产品定位

本项目定位为一套面向中国中小制造企业的开源 ERP。第一阶段优先服务离散制造企业，例如机械加工、五金、电子装配、设备零部件等行业，同时为食品、化工、原料加工等流程制造场景预留模型扩展能力。

目标用户是 20-300 人规模的中国制造企业。这类企业通常已经使用 Excel、进销存、老旧 ERP 或局部 SaaS 工具，但生产、库存、采购、质检和成本数据仍然割裂，导致库存不准、BOM 混乱、缺料停工、工单不可见和成本滞后。

本项目的差异化方向是：开源、可私有化部署、轻实施、适合中国企业使用习惯，并原生支持制造业务闭环。

## 2. 第一阶段目标

第一阶段不追求完整大 ERP，而是先打穿生产物料管控闭环。

核心目标：

1. 让物料、BOM、库存和工单数据准确起来。
2. 让销售订单能够驱动生产和采购。
3. 让生产开工前可以做齐套检查。
4. 让缺料自动形成采购建议。
5. 让工单执行过程可见。
6. 让领料、退料、补料、完工入库形成可追溯库存流水。
7. 让企业能看到基础材料成本和订单履约状态。

MVP 主线：

销售订单 -> BOM 展开 -> 齐套检查 -> 缺料采购建议 -> 生产工单 -> 领料/退料/补料 -> 工序/完工报工 -> 完工入库 -> 库存追溯 -> 基础成本。

## 3. 明确暂缓范围

以下能力第一阶段不作为主线交付，避免项目过早变成复杂 ERP：

1. 完整总账、复杂成本会计、税务申报。
2. APS 高级排程。
3. 深度 MES、设备联网、OEE。
4. PLM 和复杂工程变更管理。
5. 集团多组织、多账套合并。
6. 深度流程制造能力，例如复杂配方平衡、副产品联产、连续生产。
7. 全量 HR、CRM、OA。

这些能力需要在底层设计中预留扩展点，但不进入第一阶段核心验收。

## 4. 核心模块

### 4.1 基础资料

基础资料模块负责建立整个系统的主数据。

包含：

1. 物料档案：编码、名称、规格、分类、单位、默认仓库、采购/自制/委外属性。
2. 单位与换算：基本单位、采购单位、库存单位、生产单位。
3. 客户、供应商。
4. 仓库、库位。
5. 生产资源：工作中心、工序、班组。
6. 员工与用户。

关键要求：

1. 物料编码规则可配置。
2. 支持导入导出。
3. 主数据需要启用、停用和变更记录。

### 4.2 BOM 与工艺

BOM 是 MVP 的关键模块之一。

包含：

1. 多级 BOM。
2. BOM 版本。
3. 生效日期、失效日期。
4. 标准用量、损耗率。
5. 替代料。
6. 工艺路线和工序。
7. BOM 展开。

第一阶段的工程变更只做轻量版本：记录版本、变更说明、启停状态，不做完整 PLM 审批流。

### 4.3 库存

库存模块负责账实一致和库存追溯。

包含：

1. 入库、出库、调拨、盘点、调整。
2. 批次和序列号基础管理。
3. 现有量、已分配量、在途量、可用量。
4. 安全库存。
5. 库存流水。

库存可用量计算：

可用量 = 现有量 - 已分配量 + 可用在途量。

库存模块需要成为生产、采购、销售和成本的共同事实来源。

### 4.4 销售与需求

销售模块第一阶段只做驱动生产所需的能力。

包含：

1. 销售订单。
2. 订单交期。
3. 订单状态。
4. 销售订单转生产需求。
5. 销售订单关联生产工单、采购建议和发货记录。

暂不做完整 CRM、报价、复杂价格政策和售后服务。

### 4.5 MRP 与齐套检查

MRP 与齐套检查是第一阶段的核心价值模块。

输入：

1. 销售订单。
2. 成品或半成品 BOM。
3. 当前库存。
4. 已分配库存。
5. 在途采购。
6. 已下达生产工单。

输出：

1. 物料需求清单。
2. 齐套结果。
3. 缺料清单。
4. 采购建议。
5. 生产建议。

第一阶段使用确定性规则，不做复杂 APS。系统需要清楚告诉用户：能不能开工、缺什么、缺多少、预计什么时候可齐套。

### 4.6 采购

采购模块第一阶段围绕缺料补齐。

包含：

1. 请购单。
2. 采购订单。
3. 采购到货。
4. 采购入库。
5. 采购退货。
6. 供应商交期和默认采购价。

采购建议可以从 MRP 缺料结果生成，用户确认后转为请购单或采购订单。

### 4.7 生产工单

生产工单模块负责生产执行闭环。

包含：

1. 工单创建。
2. 工单状态：草稿、已审核、已下达、生产中、已完工、已关闭、已取消。
3. 工单用料清单。
4. 齐套检查。
5. 领料、退料、补料。
6. 工序报工。
7. 完工入库。
8. 报废记录。

第一阶段可以支持简单工序报工，不要求做到完整 MES 工位级执行。

### 4.8 质量

质量模块第一阶段只做基础质检。

包含：

1. 来料检验。
2. 生产过程检验。
3. 完工检验。
4. 不合格处理：让步接收、返工、报废、退货。
5. 检验结果关联批次、工单和供应商。

暂不做完整质量体系、SPC 和实验室管理。

### 4.9 基础成本

成本模块第一阶段不做复杂会计，只做经营可用的基础成本。

包含：

1. 标准材料成本。
2. 工单实际领料成本。
3. 报废材料成本。
4. 订单材料成本汇总。
5. 标准成本和实际成本差异。

人工、制造费用和复杂分摊规则后续再做。

## 5. 中国本地化要求

第一阶段需要原生支持中国企业的使用习惯。

包含：

1. 简体中文界面。
2. 中文单据和打印模板。
3. 人民币。
4. 中国常见税率字段。
5. 发票相关字段预留：发票类型、税号、开票信息、价税合计。
6. 审批流。
7. 数据权限：按角色、组织、仓库、单据类型控制。
8. 操作日志和审计日志。

钉钉、企业微信、飞书登录、审批和通知应作为后续扩展点设计。

## 6. 架构原则

1. 模块化：基础资料、库存、BOM、MRP、采购、销售、生产、质量、成本分模块设计。
2. 单据驱动：业务变化通过单据和状态流转表达。
3. 库存流水不可随意修改：库存数量应由业务单据生成流水，必要修正通过调整单完成。
4. 可配置：编号规则、审批流、自定义字段、打印模板、基础报表应可配置。
5. 可私有化部署：支持 Docker 和内网部署。
6. 开放 API：为移动端、扫码枪、看板、第三方系统集成预留接口。
7. 审计优先：关键单据需要记录创建、审核、反审核、作废和修改日志。

## 7. 命令行与运维工具

本项目需要设计终端能力，但终端能力不面向普通业务用户。业务用户应通过 Web UI、移动端或扫码界面完成日常操作，不能要求仓库员、采购、生产计划、车间主管、质检员和财务助理执行命令。

命令行工具面向三类用户：

1. 私有化部署人员。
2. 企业 IT 或实施顾问。
3. 开源社区开发者和二开人员。

第一阶段建议提供 `erpctl` 命令行工具，覆盖部署、运维、数据维护和开发辅助能力。

核心命令：

1. `erpctl init`：初始化环境配置。
2. `erpctl start`：启动本地或服务器环境。
3. `erpctl stop`：停止服务。
4. `erpctl status`：查看服务状态。
5. `erpctl health`：检查数据库、缓存、文件存储和后台任务状态。
6. `erpctl migrate`：执行数据库迁移。
7. `erpctl backup`：备份数据库和关键附件。
8. `erpctl restore`：从备份恢复。
9. `erpctl create-admin`：创建初始管理员。
10. `erpctl seed`：导入演示数据或行业模板数据。
11. `erpctl upgrade`：执行版本升级前检查和升级任务。

开发者命令可以由项目脚本提供，例如 `dev`、`test`、`lint`、`db:migrate` 和 `docker compose up`。后续插件体系成熟后，再增加插件开发命令，例如 `erpctl plugin create`、`erpctl plugin install` 和 `erpctl plugin publish`。

命令行设计原则：

1. 默认安全：涉及数据覆盖、恢复、清空和升级的命令需要明确确认或显式参数。
2. 可脚本化：命令应支持非交互参数，方便 CI/CD 和自动化运维。
3. 输出清晰：面向实施和运维人员，错误提示需要给出原因和下一步建议。
4. 不承载业务流程：采购、销售、工单、领料、质检等业务操作不通过 CLI 作为主入口。
5. 与 Web 管理后台互补：备份、恢复、健康检查等能力可以同时提供 Web 管理入口和 CLI 入口。

## 8. 数据流

主流程：

1. 用户维护物料、BOM、工艺、客户、供应商、仓库。
2. 销售订单创建后触发生产需求。
3. 系统根据 BOM 展开物料需求。
4. 系统结合库存、已分配量、在途量和已下达工单做齐套检查。
5. 缺料生成采购建议。
6. 用户确认采购建议，形成请购单或采购订单。
7. 到货入库增加库存和在途转现有。
8. 用户下达生产工单。
9. 工单领料减少库存并形成实际材料成本。
10. 工序报工记录生产进度。
11. 完工入库增加成品或半成品库存。
12. 发货减少成品库存。
13. 成本模块汇总标准成本与实际材料成本差异。

## 9. 错误处理与风控

关键业务规则：

1. 未审核 BOM 不允许用于正式工单。
2. 已停用物料不允许新增业务单据。
3. 库存不足时默认不允许超领，是否允许超领由系统参数控制。
4. 已关闭工单不允许继续领料、退料、报工和入库。
5. 已审核单据修改需要反审核或走变更流程。
6. 库存调整必须记录原因。
7. 批次或序列号物料必须在出入库时完整记录追溯信息。

异常提示需要面向业务用户，避免只显示技术错误。

## 10. 验收标准

第一阶段 MVP 成功的标志：

1. 可以建立一套完整的物料、BOM、仓库和客户供应商资料。
2. 可以录入销售订单并生成生产需求。
3. 可以完成 BOM 展开和齐套检查。
4. 可以根据缺料生成采购建议。
5. 可以创建、下达和执行生产工单。
6. 可以完成领料、退料、补料、报工和完工入库。
7. 可以查询库存现有量、可用量、流水和批次/序列号追溯。
8. 可以查看订单和工单的基础材料成本。
9. 可以导入导出核心主数据和单据。
10. 可以通过角色权限限制关键操作。
11. 可以通过命令行完成初始化、启动、迁移、备份、恢复、健康检查和初始管理员创建。
12. 普通业务用户可以不接触终端完成日常 ERP 操作。

## 11. 后续路线

第二阶段候选方向：

1. 更完整的财务：应收、应付、总账、凭证、月结。
2. 更完整的质量：检验方案、抽检规则、SPC、质量追溯。
3. 更完整的 MES：工位扫码、设备数据、OEE、电子看板。
4. 流程制造扩展：配方、批次投料、有效期、副产品、联产品。
5. 协同工具：钉钉、企业微信、飞书。
6. 报表与 BI：订单交付、库存周转、缺料分析、呆滞料、成本差异。
7. 插件市场和行业模板。
8. 插件开发命令和自动化部署工具。

## 12. 方案结论

该方案符合项目最初目标，但比“做一套开源 ERP”更加聚焦。项目第一阶段应避免大而全，优先解决中国中小离散制造企业最紧迫、最可验证的生产物料管控问题。

第一阶段产品成功后，再逐步扩展为完整 ERP。这样既能保持开源项目的启动可行性，也能形成与国内闭源 ERP 和国际开源 ERP 的差异化。
